灵基国内购买 · 外部系统接口需求

LBC（灵基商业中心）需要与五个外部系统对接。核心架构：LBC 不再直接调 KDM，改为 LBC → 天梯 → KDM。天梯是所有租户操作的统一编排层。KBC 与 LBC 之间需双向订单通信确保客户权益一致。灵基 KDM 产品侧前端集成 LBC 购买/增购页面。

💰 金蝶云平台（支付）

LBC 通过金蝶云平台支付能力收款，不直接对接支付宝/银行。金蝶云平台封装企业支付宝 + 对公银行转账，对 LBC 暴露统一接口。

LBC 主动调用

① 创建支付单

客户提交订单后，LBC 调此接口创建支付单，获取收银台链接。

  • 入参：订单号、金额（分，CNY）、过期时间（14天）、商品描述、回调地址
  • 出参：支付单 ID、收银台 URL / 二维码 / 银行账号
  • 触发时机：客户点"提交订单"后同步调用

② 退款接口（企业支付宝原路退）

退款审批通过后，LBC 后端调此接口发起退款。仅适用于企业支付宝支付；对公银行转账退款优先走金蝶云平台对公退款接口（⚠️ 待确认是否支持），若不支持则降级为人工线下打款。

  • 入参：原支付单 ID、退款金额、退款原因
  • 出参：退款单 ID、预计到账时间
被动接收（金蝶云平台主动推）

③ 支付 Notify 回调

金蝶云平台支付成功/失败后推送到 LBC 回调端点。开通的唯一触发源，LBC 不依赖前端回跳。

  • 包含：notify_id（幂等去重）、订单号、金额、支付方式、支付时间、签名
  • LBC 处理：验签 → 幂等 → 触发 LBC→天梯→KDM 开通链路

④ 退款成功 Notify 回调

退款成功后推送。LBC 收到后触发 LBC→天梯→KDM 租户回收流程。

  • 包含：退款单 ID、原支付单 ID、退款金额、退款成功时间
准备工作（运营配置）

⑤ 接入配置

  • 金蝶云平台支付账号开通 + 商户认证
  • 企业支付宝收款配置 / 对公银行转账收款配置
  • notify 回调端点注册 + 验签密钥获取
  • 退款接口权限申请

🔵 金蝶云平台（企业身份）

金蝶云账号体系。提供 TID（企业唯一标识）、企业认证状态、云账号注册能力。LBC 只读引用，不重复做认证。

LBC 主动调用

① 用手机号/邮箱查企业信息（全量含认证状态）

试用留资、官网新购时，LBC 用客户手机号/邮箱查询该用户关联的企业信息。

  • 入参：手机号 或 邮箱
  • 出参：TID、企业名称、统一社会信用代码、认证状态（certified / uncertified / none）、关联账号列表
  • 用途：试用需要、官网新购需要
  • 待确认：一个手机号可能对应多个企业，返回列表还是最近活跃的？

② 用 TID 查企业信息（全量含认证状态）

增购、产品内新购时，LBC 用已知 TID 查询企业最新信息和认证状态。

  • 入参：企业 TID
  • 出参：TID、企业名称、统一社会信用代码、认证状态、管理员信息
  • 用途：增购需要、产品内新购需要

③ 代注册云账号接口

客户无金蝶云账号，LBC 用留资手机号代注册云账号。试用需要、官网新购需要。

  • 入参：手机号、联系人姓名
  • 出参：云账号 ID
  • 待确认：云平台是否开放此接口？若不支持，备选方案为生成注册链接引导客户自己完成

④ 代注册企业接口

客户有云账号但无企业，LBC 代在已有账号下注册企业，返回企业 TID（未认证状态）。试用需要、官网新购需要。

  • 入参：云账号 ID、企业名称、联系人信息
  • 出参：企业 TID（未认证状态）
  • 待确认：注册企业后是否立即返回 TID？TID 是否需要认证完成才有？

⑤ 刷新/查询认证状态

客户已支付但企业认证未完成（pending_activation），LBC 轮询认证状态，完成后自动触发开通。

  • 入参：企业 TID
  • 出参：当前认证状态、认证完成时间
核心待确认项：① 代注册接口（云账号+企业）是否开放；② 手机号/邮箱查企业的接口格式和多企业处理逻辑。认证完成由 LBC 主动轮询"刷新/查询认证状态"接口（⑤）获知，不依赖云平台回调。

🟣 KBC（SLG 系统 + 财务 + 发票 + 合同）

KBC 是 SN 定义方、发票开具方、财务记账执行方、电子签提供方。LBC 开通前同步获取 SN，开通后异步推财务事件/开票/签约。KBC 与 LBC 之间需要双向订单通信，确保客户权益一致、续费不丢。

LBC 主动调用 — 开通前（同步，关键路径）

① 查询 SN（以 TID 查该客户是否已有灵基 SN）

开通前同步调用，判断该客户是新购还是增购/续费。关键路径，SN 必须在调天梯前就位。

  • 入参：客户 TID、产品编码（LINGEE_PRO / LINGEE_ULTRA）
  • 出参：是否存在 SN、SN 值（存在时返回）

② 生成 SN（新购时同步生成新 SN 并返回）

新购场景，KBC 同步生成新 SN 并返回给 LBC。同一 TID + 同一产品，SN 唯一且不变。

  • 入参：客户 TID、产品编码、客户企业信息
  • 出参：新 SN
  • 超时处理：超时 → LBC 进待开通队列，客户看到"处理中"
LBC ↔ KBC 双向订单通信（确保权益一致）

③ 查询灵基订单列表（含退款订单）

KBC 需要查到 LBC 侧的灵基订单信息，确保续费时权益一致。

  • 待确认：KBC 主动查 LBC？还是 LBC 推给 KBC？方案待和 KBC 团队沟通
  • 数据：订单号、产品版本、席位数、点数、服务期、退款状态

④ 推送灵基订单列表（含退款订单）

LBC 把灵基订单（新购/增购/续费/退款）推给 KBC，解决权益不一致问题。

  • 入参：TID、SN、订单列表（含订单类型/金额/服务期/状态）
  • 出参：接收确认
  • 待确认：推送频率（实时事件推送 vs 定时批量同步）？接口格式？
LBC 主动调用 — 开通后（异步旁路，不阻断开通）

⑤ 开票接口（增值税电子发票，普票/专票）

开通成功后异步调用，KBC 开具增值税电子发票。

  • 入参：购买方企业名称、统一社会信用代码、含税金额（分）、税率、商品明细、发票类型（general / special）
  • 出参：发票号、PDF 链接、开票时间

⑥ 重开发票（客户修改开票信息）

客户申请修改开票信息，LBC 调此接口作废原发票并重开。

  • 入参：原发票号、新开票信息（企业名称/税号等）
  • 出参：新发票号、新 PDF 链接

⑦ 红字发票（退款红冲）

退款成功后，LBC 通知 KBC 开具红字发票。

  • 入参：原发票号、退款金额、退款原因
  • 出参：红字发票号
  • 触发时机：退款成功回调后异步

⑧ 单签合同（电子签盖章）

开通成功后，LBC 生成订购确认函 PDF，调 KBC 提供的电子签接口加盖金蝶电子公章。KBC 已集成法大大/上上签。

  • 入参：PDF 文件/URL、盖章位置信息、金蝶公章 ID
  • 出参：带章 PDF 下载链接
KBC 核心待确认项：① 双向订单通信方案（KBC 查 LBC 还是 LBC 推 KBC）；② 开票/重开/红冲 API 格式；③ 电子签接口文档。

🔧 金蝶天梯（统一编排层）

天梯是所有租户操作的统一编排层，LBC 通过天梯调度 KDM。天梯接收 LBC 指令后：① 透传 TID+SN+许可给 KDM；② 查集群信息（ERP 集群连接）；③ 准备物理资源（专属云模式）；④ 调 KDM 执行开通/增购/续费/回收。不只是 SLG 链路，自助购买链路也走天梯。

⚠️ 待跨部门澄清：以下 5 个操作是一个统一 API 传不同 action_type，还是 5 个独立 API？入参/出参格式需与天梯团队对齐。
LBC → 天梯 → KDM（统一租户操作）

① 开通付费租户

LBC→天梯→KDM：建生产+沙箱租户，开权益，派发点数。新购支付+TID/SN 就位后触发。

  • 入参：action_type=provision_paid / TID / SN / 产品版本 / 席位数 / 赠送点数配置 / 订阅期
  • 出参：执行状态 / 生产租户 URL / 沙箱租户 URL / 失败原因
  • 天梯加工：查集群信息、分配资源后调 KDM 执行

② 开通试用租户

LBC→天梯→KDM：建 trial 租户，带 TID。试用邀请码匹配后触发。

  • 入参：action_type=provision_trial / TID / 产品编码 / 试用期限
  • 出参：执行状态 / trial 租户 URL / 试用到期时间 / 失败原因

③ 增购权益激活

LBC→天梯→KDM：追加席位/注入点数。增购支付+对账通过后触发。

  • 入参：action_type=expansion / TID / SN / 追加席位数（席位增购时）/ 点数金额（点数增购时）/ expires_at
  • 出参：执行状态 / 当前总席位数 / 当前点数余额 / 失败原因

④ 续费权益激活

LBC→天梯→KDM：延长 term_end + 重置赠送点数周期。续费支付后触发。

  • 入参：action_type=renewal / TID / SN / 新 term_end / 席位数（可能同时加席位）
  • 出参：执行状态 / 新 term_end 确认 / 失败原因

⑤ 触发租户回收

LBC→天梯→KDM：租户 suspended→grace_period→terminated，点数清零。退款成功后触发。

  • 入参：action_type=reclaim / TID / SN
  • 出参：执行状态 / 回收状态 / grace_until 时间 / 失败原因
统一入参结构：action_type / TID / SN / 产品版本 / 席位数 / 点数 / term_end 等。统一出参结构：执行状态 / 租户 URL（新开通时）/ 失败原因。天梯负责编排，LBC 只需发指令，不关心底层资源调度细节。

🟦 灵基 KDM（产品侧）

灵基 KDM 产品侧有入口唤起 LBC 的购买/增购页面（前端集成）。购买完成后前端关闭弹窗刷新界面，正式订单/权益推送走后台完整流程（LBC→天梯→KDM）。

前端集成（非后端 API）

① 产品内唤起增购页

灵基 console 内"增购席位"/"购买点数"按钮，打开 LBC 增购页面。前端集成，非后端 API。

  • 方式：弹窗（iframe）或跳转 URL
  • 传参：TID、SN、当前产品版本、当前席位数、来源标识
  • 场景：已有付费租户的客户在产品内追加席位或购买点数

② 产品内唤起新购页

灵基 console 内"升级/购买"按钮，打开 LBC 新购页面。前端集成，非后端 API。

  • 方式：弹窗（iframe）或跳转 URL
  • 传参：TID、当前试用状态、来源标识
  • 场景：试用客户升级为付费、或免费功能引导购买

③ 增购/新购完成回调（前端交互）

购买完成后 LBC 通知灵基产品侧"关闭弹窗刷新界面"。此处仅前端交互，正式权益推送走后台流程（LBC→天梯→KDM）。

  • 方式：前端 postMessage 或回调 URL 带参数
  • 传递信息：购买完成状态、订单类型（新购/增购）
  • KDM 前端行为：关闭弹窗、刷新权益展示界面
设计要点：前端集成使购买体验无缝（不离开灵基产品），正式权益变更走后台完整链路（LBC→天梯→KDM）保证数据一致性。前端只负责"关闭弹窗+刷新UI"，不承担权益写入。

接口需求汇总

共 22 个接口/集成需求，按系统汇总。核心架构：LBC→天梯→KDM（不再直连）；LBC↔KBC 双向订单同步；KDM 前端集成 LBC 购买页。
