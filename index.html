0业务背景与目标

做什么：灵基（Lingee）国内自助购买站点 —— 选产品 → 填信息 → 下单支付 → 自动开通环境。

核心诉求：① 客户体验顺畅、自助完成；② 支付与开票合规；③ 支付后自动开通产品；④ 金蝶云平台处理收款。

  • 两个版本：专业版（自助，¥99/席/月，起售 5 席）、高级版（面议，起售 100 席，多 AI 财务部 + CEO 办公室治理能力）
  • 点数：阶梯定价 ¥0.18–0.20/点，最小 1 万点；每席每月赠送 500 点（仅生产环境，当月有效）
  • 每套产品：1 生产租户 + 1 沙箱租户（隔离），公有云；可连客户私有部署 ERP（星瀚/星空）
  • 收款：金蝶云平台（支付能力，封装企业支付宝 + 对公银行转账），人民币定价与结算
  • 发票：LBC 调用 KBC 开票接口，KBC 开具增值税电子发票，LBC 存储并推送客户
主数据源（SoT）原则：灵基商业化后台（LBC）是订单/订阅/席位/点数/权益的权威源；金蝶云平台是支付执行层（支付结果/流水以其回调为准）；KBC 是 SLG 业务系统与金蝶财务系统（开票/收入确认/ERP核算）的权威源；TID（客户主数据）与企业认证由金蝶云账号体系管理；金蝶云平台（企业身份）提供企业认证能力（LBC 只引用，不重复做认证）。

1.2 灵基点数规则
  • 赠送：每席每月 500 点 → 仅生产环境、按月派发、当月有效、过期作废、不结转
  • 增购：阶梯定价 ¥0.20/点（1–50 万点）/ ¥0.19/点（51–100 万点）/ ¥0.18/点（101 万点以上），最小 1 万点 → 生产 + 沙箱均可用、可分配到人（个人额度上限内共享）、即时发放、随主合同租期有效、过期作废
  • 记账：点数用流水账本（POINT_LEDGER），赠送/增购分别按 expires_at 失效；余额由账本汇总
1.3 购买旅程

A. 专业版自助新购（PLG，无销售）：

选产品/配置 → 企业认证检测 → 勾选服务协议（同意存证）→〔点提交订单：建待支付订单〕→ 金蝶云平台收银台
→ 支付成功（notify 回调）→ 履约对账 → TID/SN 前置闸（TID 已有；SN 同步从 KBC 获取/生成）→ 开通中（建生产+沙箱租户/权益/点数）→ 开通成功
→ 异步旁路：KBC 开票 + 电子签出订购确认函 + 推送邮件

B. 增购席位（在用正式客户，产品内）：

产品内「增购席位」入口 → 填写席位数 → 勾选协议（再次存证）→ 建增购订单（order_type=expansion）
→ 金蝶云平台收银台 → 支付成功（notify 回调）→ TID/SN 对齐（SN 不变，挂已有租户）
→ KDM 追加席位 + 追加赠送点数配额 → 开通成功 → 异步：KBC 开票 + 电子签出确认函 + 推送邮件

C. 增购点数（在用正式客户，产品内）：

产品内「增购点数」入口 → 选购点数档位（阶梯定价）→ 勾选协议 → 建增购订单（order_type=add_on）
→ 金蝶云平台收银台 → 支付成功（notify 回调）→ 验签/幂等/金额匹配
→ KDM POINT_LEDGER 注入购买点数（生产+沙箱均可用，随主合同租期有效）
→ 异步：KBC 开票 + 电子签出确认函 + 推送邮件

D. 续费（订阅到期前）：

续费提醒（到期前 30/15/7 天邮件通知）→ 客户点「续费」→ 确认席位数（可同时加席位）
→ 勾选协议 → 建续费订单（order_type=renewal）→ 金蝶云平台收银台
→ 支付成功（notify 回调）→ TID/SN 对齐（SN 不变）→ KDM 延长租期 + 重置赠送点数周期
→ 异步：KBC 注册续费 SN + KBC 开票 + 电子签出确认函 + 推送邮件

E. 高级版/大客户面议（销售漏斗）：

线索录入 → 销售跟进 → 商机报价 → 赢单付款 → 转成 ORDER → 开通 → SUBSCRIPTION/TENANT

F. 邀请码试用 → 新购（PLG，落地页留资）：

落地页填表（姓名/手机号/公司名/邀请码）→ 以手机号查金蝶云平台
· 老客户（已有企业TID）→ 自动关联TID
· 新客户（无企业TID）→ 代注册云账号+企业（⚠️接口可行性待确认） → 获得企业TID（未认证状态）
→ LBC 校验邀请码：匹配 → 调天梯→KDM 建 type=trial 试用租户（带TID）→ auto_activated
→ 码不匹配/KDM 失败 → failed，运营人工审核 → manual_activated
→ 客户在产品内点「升级购买」→ 进入购买页（流程同A，Section 3 解耦流程）
→ 新购开通成功 → 试用租户 type 改为 sandbox（正式沙箱，数据保留）+ 单独新建生产租户
  • 建单时机：勾选协议并点「提交订单」即创建订单（pending_payment）并落同意存证，再创建金蝶云平台收银台
  • 支付：企业支付宝（扫码，异步 notify 回调）或对公银行转账（notify 回调，T+0 至 T+3 天）
  • 开通：由金蝶云平台支付 notify 回调异步驱动，先过 TID/SN 对齐闸再调天梯→KDM 开通
  • 协议：增购/点数购买/续费均须再次勾选存证；增购/续费适用原订阅协议，无需重签
增购入口汇总（产品内）：
① 席位增购：管理员在「席位管理」页或套餐页点「增购席位」→ 建 order_type=expansion 订单
② 点数增购：管理员或用户在「点数余额」页点「购买点数」→ 建 order_type=add_on 订单
③ 续费：到期提醒弹窗 / 邮件中的「立即续费」链接 → 建 order_type=renewal 订单
三种增购场景的 SN 均不变，挂已有租户，不重新开通；TID/SN 对齐闸做防重开检查。
线索来源（国内扩展）：官网询价 / 产品内询价 / 试用申请 / 销售手工录入 / 线下活动 / 老客户（存量/转介绍）。与海外版相比，多了「线下活动」和「老客户」两个来源，系统均以 source 字段区分。
1.4 订单状态 & 状态页规范

订单状态（ORDER.status，10 值，核心 9 态与海外版完全一致，国内新增 pending_activation）：

状态含义下一步引导
pending_payment待支付（14 天有效）继续支付 / 稍后 / 取消
under_review已付款，履约对账+风控检查中等待，通知激活
pending_activation【国内特有】已付款+对账通过，但 TID 缺失（企业认证未完成），等待认证后自动开通完成企业认证（含认证跳转链接）/ 申请退款
provisioning履约中（建租户/加席位/续期）等待（自动跳完成）
completed已完成前往灵基
rejected风控拒绝（黑名单命中）联系客服，全额退款
activation_failed已付款但开通失败联系客服，补开或退款
cancelled客户主动取消重新下单
closed超时未付自动关闭重新下单
refunded已退款（+ 环境回收）—

国内特有：① 对公银行转账在 under_review 状态文案改为「正在核对银行流水，通常 1–3 个工作日」；② pending_activation 为国内版新增状态——已付款但企业认证未完成时进入，认证完成后自动触发 → provisioning，期间客户可申请退款（已付款未交付，我方责任）。

1.5 合同与协议

专业版自助购买：格式合同（单签），无金额上限限制。

  • 客户在购买页勾选《灵基服务订阅协议》即构成有效合同（依据《民法典》合同编 + 《电子商务法》），无需双方签名
  • 每次新购、增购均须再次勾选存证，证明客户知悉并同意当前版本协议
  • 增购适用原订阅协议，无需重新签合同

订购确认函（电子签章，甲方单签）：

  • 开通成功后，LBC 自动生成订购确认函 PDF，调金蝶云平台或 KBC 提供的电子签接口（已集成法大大 / 上上签，具体平台待运营确认）为金蝶加盖电子公章，生成带章文件
  • 带章 PDF 由 LBC 存储，并通过邮件推送给客户；同一封邮件附增值税电子发票链接 + 所同意协议版本链接
  • 订购确认函包含：合同编号 / 甲方主体（金蝶软件（中国）有限公司）/ 乙方主体（客户企业名称 + 统一社会信用代码）/ 产品版本 / 席位数 / 订阅周期 / 含税金额 / 协议版本号 + 链接 / 开通时间 / 金蝶电子公章
  • 客户无需回签；此函作为客户财务报销 / 采购系统的合同附件
  • 退款时函件不作废（合同已履行），但退款确认邮件中注明服务终止信息

为什么要做单签确认函：企业客户（尤其金蝶 ERP 生态客户）财务规范度较高，采购系统普遍要求"有公章的合同文件"作为付款凭证；纯勾选存证在法律上有效，但在客户内部报销流程上会产生摩擦。单签确认函解决这一问题，且系统自动生成，不增加人工成本。

协议版本管理：

  • 协议正文由金蝶法务系统统一存储与维护，LBC 只引用外部链接
  • LBC 维护协议版本记录（版本号 / 外部链接 / 是否当前有效 / 生效日期）
  • 状态流转：draft（草稿，可修改）→ active（生效，字段冻结）→ archived（归档，只读永久保留）
  • 有同意存证绑定的版本记录永远不能删除，所有操作写 AUDIT_LOG

同意存证记录字段：

  • account_id / contact_email（谁）
  • agreement_version_id（哪个版本）
  • page_enter_time（进入协议页时间）
  • checkbox_checked_time（勾选时间）
  • order_created_time（下单时间）
  • ip_address / user_agent / session_id（操作环境）

时序要求：page_enter_time < checkbox_checked_time < order_created_time < paid_at，完整时序链证明客户有机会阅读并主动同意。电子签盖章在开通后异步触发，不在支付关键路径。

1.6 合规要求（国内版）
维度要点
企业认证
（购买时机）		 法律要求：发票需要企业名称 + 统一社会信用代码，合同主体需确定，但法律没有规定认证必须在支付前完成。
产品设计决策（支付前完成）：不是法律强制，而是体验选择——付款后才发现未认证会阻断开通，是极差的客户体验；在支付前检测并引导，即使多一步，体验可控。
执行：由金蝶云平台（企业身份）提供认证能力，LBC 只读引用（云账号体系：TID · 企业名称 · 统一社会信用代码 · 认证状态）
合同/协议格式条款单签，客户勾选即有效；协议正文存法务系统，LBC 引用链接；同意存证 LBC 自管；开通后自动生成带金蝶电子公章的订购确认函 PDF（甲方单签），通过金蝶云平台或 KBC 提供的电子签接口盖章（已集成法大大/上上签，具体待确认），推送客户作为合同凭证
发票LBC 调 KBC 开票接口 → KBC 开具增值税电子发票 → LBC 存储并推送客户
个人信息保护
（PIPL）		 法律依据：《个人信息保护法》（2021）要求处理个人信息须遵循最小必要原则（只收集与处理目的直接相关的最少信息），并取得明示同意或具备其他合法依据。

国内行业实践：B2B SaaS 普遍在注册/购买环节弹出《隐私政策》勾选同意，不强制收集与业务无关字段（如身份证号），企业认证信息依托平台方（本项目依托金蝶云账号体系）。

LBC 的具体义务：
  • 购买页仅收集：企业名称 / 统一社会信用代码（来自金蝶云账号认证，不重复收集）/ 联系人姓名 / 手机号 / 邮箱 / 发票抬头信息
  • 数据驻留国内，不出境（国内版与海外版分开部署）
  • 保留期：订单/发票/同意存证按法律/监管要求保留（通常 5 年）；租户数据回收后 30 天保留期到期删除/匿名化
  • 展示：购买页须明示《隐私政策》链接，开通邮件注明数据用途
反欺诈 &
本地黑名单		 法律依据：国内无强制反欺诈合规名单筛查义务（不同于海外 OFAC/制裁名单），但《电子商务法》要求平台对欺诈交易有防控措施。

国内行业实践：B2B SaaS 通常维护内部黑名单（欺诈/拒付/违规封禁账户），无需接入第三方 OFAC/出口管制名单。

LBC 的具体设计：本地黑名单（运营手工维护封禁账户）+ 基础反欺诈（异常金额/频率告警）。无制裁名单、无出口管制要求。
网络安全 &
ICP		 法律依据：《网络安全法》（2017，2026 年修订版已生效）要求互联网信息服务须持有 ICP 许可证；关键信息基础设施运营者须将数据存境内。

LBC 适用情况：ICP 许可由金蝶软件（中国）有限公司持有，灵基国内版作为其服务在证内；数据驻留国内已满足；买方无强制实名要求（企业云账号认证已覆盖）。

国内版无 OFAC/制裁名单/出口管制义务，这是与海外版的核心合规差异。国内合规重心在 PIPL（个人信息保护）、ICP（网络安全）、增值税发票（财税）三个维度。

2系统设计
2.1 架构与边界（三层）
  • SoT 灵基 Lingee BC（商业中心/商务层）：产品/SKU、订单、订阅、协议存证、notify 回调、TID/SN 前置闸、通知、工单、审批、审计、电子签确认函
  • 灵基 KDM（产品/履约层）：建生产+沙箱租户、权益开关、点数派发与计量、ERP 连接器、集群连接注册表
  • 金蝶天梯（统一编排层）：所有租户操作的统一编排层（自助+SLG 均走天梯），透传+加工：查集群信息、准备物理资源后再转 KDM
  • 金蝶云平台（支付） 支付执行层：企业支付宝 + 对公银行转账，统一 notify 回调，CNY 结算
  • 金蝶云平台（企业身份） 云账号体系：TID 主数据、企业认证、统一社会信用代码，LBC 只读引用；支持代创建云账号/注册企业（接口可行性待确认）
  • KBC（SLG + 财务）：SLG 销售系统、SN 生成与管理、收款记账、收入确认、ERP 核算
  • KBC（发票）：增值税电子发票开具（普票/专票）、红字发票（退款红冲）、重开、作废

支付成功 ≠ 立即开通：notify 回调 → 履约对账（流水↔订单匹配） → TID/SN 前置闸（TID 支付前已有或认证后补入；以 TID 查 KBC 同步获取 SN，新购时 KBC 生成新 SN，增购时返回已有 SN；TID/SN 均就位才调天梯→KDM 开通）→ 开通后异步推事件给 KBC（发票 + 财务记账）。履约对账在 LBC、秒级；财务记账在 KBC、开通后异步推事件、不阻断开通。

2.3 支付与对账
  • 支付集成：接金蝶云平台支付能力，不直接接支付宝/银行。金蝶云平台封装企业支付宝（扫码）和对公银行转账，对 LBC 暴露统一的创建支付单 API 和 notify 回调格式
  • 企业支付宝：扫码即支付，支付成功后金蝶云平台推送 notify，通常秒级到达
  • 对公银行转账：客户手动转账，银行流水回来后金蝶云平台推送 notify，T+0 至 T+3 天
  • notify 处理：验签 + 幂等（按 notify_id 去重）+ 失败重试，与海外 Webhook 处理逻辑完全对称

两层对账（与海外版一致）：

  • 履约对账（开通前，关键路径，秒级）：金蝶云平台支付回调金额 ↔ 订单金额匹配，匹配才触发开通
  • 财务对账（开通后，异步，不阻断）：LBC 订单 ↔ 金蝶云平台流水 ↔ KBC 账务，后台提供对账视图

退款：

  • 企业支付宝：调金蝶云平台退款接口，原路退回，通常 T+1 到账
  • 对公银行转账：客户提供退款账户信息 → LBC 调金蝶云平台对公退款接口（⚠️ 待确认云平台是否支持；若不支持则降级为人工线下打款）
  • 退款成功后：异步推 KBC 做收入冲减（已入总账需冲减）+ 通知 KBC（发票）开红字发票
  • 开通后退款需人工双人审批（同海外版）
2.4 发票
  • 开票流程：开通完成后，LBC 调用 KBC 开票接口（传入：购买方企业名称/统一社会信用代码/金额/税率/明细）→ KBC 开具增值税电子发票（普票/专票）→ 返回发票文件/链接 → LBC 存储并通过通知服务推送客户
  • LBC 存什么：KBC 返回的发票号 / 发票文件链接 / 开票时间 / 发票类型（普票/专票）/ 状态
  • LBC 不存什么：税控证书、开票业务逻辑、发票正文（均在 KBC）
  • 重开发票：客户在管理后台可申请修改开票信息（企业名称/税号填错），LBC 再次调 KBC 接口重开，旧发票在 KBC 作废
  • 退款发票：退款成功后 LBC 通知 KBC 作废原发票，KBC 开具红字发票，LBC 更新发票状态
2.5 开通与履约
  • 金蝶云平台 notify → 验签/幂等 → 履约对账（金额匹配） → TID/SN 前置闸：
    • TID：支付前企业认证步骤已得（来自金蝶云平台企业身份），到此已就绪；若未完成认证，支付对账通过后进入 pending_activation，认证完成后自动补入 TID 继续开通
    • SN：以 TID 调 KBC 同步获取——有 SN 则增购挂已有租户；无 SN 则 KBC 同步生成新 SN 并返回；KBC 超时进待开通队列
    • TID + SN 均到位后才调天梯→KDM 开通，不能只有 LBC 自生成的临时 SN 开通后再补（已开通租户的敏感身份信息不可事后修改）
  • KDM：建 1 生产 + 1 沙箱租户（隔离）→ 按版本开权益 → 派发赠送点数 + 注入增购点数 + 启用计量
  • 赠送点数：按月定时任务派发 500/席，月底清零不结转
  • 开通失败 → activation_failed（已付款，补开或退款）
  • 开通成功后通过订单同步将最终状态订单推送 KBC，KBC 据订单状态做收款记账 + 收入确认（不阻断开通）
  • 退款后环境回收：同海外版（SUBSCRIPTION 取消 → TENANT suspended → 保留期 30 天 → terminated → 点数清零）

⚠️ 海外版已知卡点（国内版不受影响）：KBC 生成 SN 接口及金蝶云平台企业身份接口的海外版计划年底建设完成。国内版接口现在就有，可直接按 Phase 1 目标态实现（TID 支付前从企业认证得到，SN 开通前同步从 KBC 获取）。海外版在接口就绪前为 Phase 0 过渡状态，就绪后需做存量 SN 迁移对齐。

3.3 研发交接
  • 建单时机：勾选协议并点「提交订单」→ 建 pending_payment 订单 + 落同意存证（含 page_enter_time/checkbox_checked_time/IP/UA/session_id）→ 创建金蝶云平台收银台
  • notify 回调：验签 + 幂等（notify_id 唯一约束）+ 重试，是开通的唯一触发源，禁止在前端回调里开通
  • TID/SN 前置闸：TID 在支付前企业认证时已就绪；SN 在 notify 回调后、开通前同步从 KBC 获取（新购时 KBC 同步生成新 SN，增购时返回已有 SN）——两者必须在调天梯→KDM 开通前都持有，不允许开通后再补（已开通租户的身份信息不可事后修改）。Phase 0 KBC 接口未就绪时用 LBC 临时 SN，是已知技术债
  • 试用升级：新购开通成功后，试用租户 type 改为 sandbox（正式沙箱，数据保留），同时新建独立的正式生产租户
  • 协议版本：前端读 LBC 最新 is_current=true 的版本链接，不硬编码
  • 发票：开通成功后异步调 KBC 开票接口，不在开通关键路径上
  • 所有金额用整数（分）存储，展示时除以 100
